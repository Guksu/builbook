# 웹소설 에디터 제품 스펙 작성

스크리브너를 레퍼런스로 하되, **진입장벽을 극단적으로 낮춘** 웹소설 집필 도구의 제품 스펙을 작성한다. 핵심 원칙: 입문 작가가 가입 후 5분 안에 첫 문장을 쓸 수 있어야 한다.

## 스크리브너 본질의 단순화 매핑
스크리브너 기능을 그대로 베끼면 진입장벽이 올라간다. 본질만 남기고 번역한다:

| 스크리브너 개념 | 본질 | 입문자용 단순화 |
|----------------|------|----------------|
| Binder | 문서 트리 | 폴더/문서 2종만. 드래그로 정렬. 우클릭 메뉴 최소화 |
| Corkboard / Outliner | 구조 조망 | MVP에서 제외 또는 단순 목록 |
| Inspector(메타데이터) | 문서별 메모·시놉시스 | 접힌 사이드 패널 1개. 기본은 숨김 |
| Compile(내보내기) | 원고 출력 | MVP는 단순 텍스트/마크다운 내보내기만 |
| Collections/Labels | 분류 | MVP에서 제외 |

## 작성 절차
1. **MVP 범위 결정** — 빼는 것부터 정한다. "이 기능이 입문 작가를 멈칫하게 하는가?"를 기준으로 거른다. 멈칫하면 기본값 뒤로 숨기거나 백로그로 분리.
2. **IA·화면 흐름** — 화면 목록과 이동 흐름. **빈 상태부터** 설계한다(작품 0개, 문서 빈 화면). 첫 경험이 진입장벽을 좌우한다.
3. **기능별 스펙** — 각 기능마다 목적·동작·상태·엣지케이스. 상태는 반드시 명시(예: 저장 안 됨 → 저장 중 → 저장됨). 상태 누락은 후속 경계면 버그의 씨앗.
4. **도메인 모델 확정** — 아래 표준 모델을 스펙에 포함하고, 변경 시 data-modeler에 통지.

## 표준 도메인 모델 (전 에이전트 공유)
- **User** — 작가 계정
- **Project** — 하나의 작품. title, description, owner
- **Document** — 바인더 트리 노드. parentId(자기참조), type(FOLDER|DOC), title, order, content(Tiptap JSON), synopsis, wordCount
- **Character** — 인물 카드. projectId, name, description, fields(JSON)
- **Snapshot** — 문서 버전 기록. documentId, content, createdAt

## 출력 형식: `_workspace/01_product_spec.md`
```
# 제품 스펙: {제품명}
## 1. 비전 & 진입장벽 전략
## 2. MVP 범위 (포함 / 제외 / 백로그)
## 3. 정보구조 & 화면 목록
## 4. 화면 흐름 (빈 상태 우선)
## 5. 기능별 스펙 (목적/동작/상태표/엣지케이스)
## 6. 도메인 모델
```

## 핵심 원칙
- **빼는 결정이 넣는 결정보다 중요하다.** 단순함이 이 제품의 경쟁력이다.
- 모든 데이터 화면에 로딩·에러·빈 상태 3가지를 정의한다.
- 상태와 전이를 명시해 후속 에이전트가 추측하지 않게 한다.
