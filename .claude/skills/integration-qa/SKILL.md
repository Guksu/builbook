# 통합 정합성 검증 (Integration Coherence)

개별 모듈이 각자 정상이어도 **연결 지점(경계면)에서 계약이 어긋나** 런타임 크래시가 난다. `npm run build` 통과는 정상 동작을 보장하지 않는다 — 제네릭 캐스팅·`any`가 타입 안전성을 우회하기 때문. 이 스킬은 "존재 확인"이 아니라 **"교차 비교"** 를 한다.

## 검증 방법: 양쪽 동시 읽기
생산자와 소비자를 **같이 열어** 비교한다. 한쪽만 읽으면 못 잡는다.

### 1. API 응답 shape ↔ 프론트 훅 타입
```
1. 모든 route.ts에서 NextResponse.json(...)의 인자 shape 추출 (Grep: "NextResponse.json")
2. 대응 훅에서 응답을 어떻게 쓰는지 확인 (배열로 .filter/.map 하는지, .items를 꺼내는지)
3. 컬렉션이 { items: T[] }인데 훅이 배열로 바로 쓰면 → 런타임 크래시. 적발.
4. 단일/컬렉션 구분, 래핑 unwrap 여부, camelCase 일치 확인
```

### 2. 라우팅 경로 ↔ 링크
```
1. app/ 하위 page.tsx 경로에서 URL 패턴 추출 ((group) 제거, [param] 동적)
2. 코드 내 모든 href=, router.push(, redirect( 값 수집 (Grep)
3. 각 링크가 실제 page 경로와 매칭되는지 확인 → 안 맞으면 404 버그
```

### 3. 상태 전이 ↔ 코드
```
1. 저장 상태(idle/saving/saved/error) 등 상태 정의 추출
2. 모든 status/state 업데이트 코드 검색
3. 정의됐으나 코드에서 도달 불가한 죽은 상태, 코드엔 있으나 정의 안 된 무단 상태 적발
```

### 4. API ↔ 훅 1:1 매핑
```
1. app/api/ 엔드포인트 목록 추출
2. hooks/의 fetch URL 목록 추출
3. 호출 안 되는 API(훅 누락), 존재 안 하는 API 호출 적발
```

### 5. 자동저장 경계면
```
에디터의 PUT content 요청 body shape ↔ route.ts가 기대하는 body가 일치하는지 확인
```

## 점진 검증 (Incremental QA)
전체 완성 후 1회가 아니라, **각 모듈(특히 백엔드 API) 완성 직후** 해당 API+대응 훅을 즉시 교차 검증한다. 초기 불일치가 후속 모듈로 전파되는 것을 막는다.

## 출력: `_workspace/07_qa_report.md`
```
## 통합 정합성 검증 리포트
### ✅ 통과
- [항목] API↔훅 shape 일치
### ❌ 실패
- [경계면] 파일:라인 — 문제 — 수정 방법 — 담당 에이전트
### ⚠️ 미검증 / 확인 필요
- [항목] 의도된 설계인지 불명확
```

## 핵심 원칙
- 약한 체크리스트("API가 존재하는가?")가 아니라 강한 체크리스트("응답이 호출측 기대와 일치하는가?")를 쓴다.
- 발견 즉시 양쪽 담당 에이전트에 `파일:라인 + 수정 방법`으로 통지.
- 의도/버그가 모호하면 삭제·단정하지 말고 "확인 필요"로 분류해 질의.
