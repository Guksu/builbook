# 원티드 디자인 시스템 적용

원티드 디자인 시스템을 기반으로 일관된 토큰과 컴포넌트를 구축한다. 토큰을 단일 진실 공급원(SSOT)으로 두어, 색·간격·폰트가 코드 곳곳에 하드코딩되는 것을 막는다 — 하드코딩은 디자인 일관성 붕괴와 변경 비용 폭증의 원인이다.

## 워크플로우

### 1단계: 토큰 검증
`references/wanted-tokens.md`를 읽는다. 이 문서의 컬러·타이포·스페이싱 값 상당수는 `[추정]`이다. **추정값을 무비판적으로 쓰지 않는다** — 잘못된 브랜드 컬러로 전체 UI를 칠하면 나중에 전부 교체해야 한다.

- 네트워크가 허용되면 레퍼런스 "부록: 검증 체크리스트"를 수행해 실제 원티드 값(특히 시그니처 블루 hex)으로 교체한다.
- 네트워크가 차단됐거나 확인 불가하면 추정값을 쓰되, 산출물(`globals.css` 주석, `06_design_system.md`)에 "추정 토큰 — 검증 필요"를 명시한다. 거짓 확신보다 정직한 표기가 후속 수정을 쉽게 한다.

### 2단계: 폰트 적용
wanted-sans를 적용한다. 기본은 `next/font/local`(self-host) — 외부 의존과 layout shift가 줄어든다. 빠른 프로토타이핑이면 CDN `@import`도 가능. 폰트 파일 경로는 설치된 패키지 실제 dist 구조로 검증한다(레퍼런스의 예시 경로는 검증 전 가정값). fallback 스택은 한글 환경(Apple SD Gothic Neo/Pretendard/Noto Sans KR)을 반드시 포함한다.

### 3단계: 토큰 구축
`references/wanted-tokens.md`의 `:root` 블록을 `app/globals.css`에 통합하고, Tailwind를 쓰면 `theme.extend`가 CSS 변수를 참조하게 매핑한다(레퍼런스 6-2). CSS 변수가 원본이고 Tailwind는 그것을 가리키는 별칭이어야 한다 — 두 곳에 값을 중복 정의하면 어긋난다.

### 4단계: 기본 컴포넌트
토큰만 사용하는 기본 컴포넌트를 만든다 — Button(primary/secondary/ghost), Input, Textarea, Card, Modal, Dropdown, Toast. 진입장벽 낮은 제품이므로 친근하고 단순하게: 과한 그림자·강한 대비·복잡한 상태를 피한다. 접근성 기본(대비비, 포커스 링, 키보드 조작)을 지킨다.

### 5단계: 일관성 검수
다른 에이전트가 만든 화면에서 토큰을 우회한 하드코딩을 Grep으로 적발한다:
- `#[0-9a-fA-F]{3,6}` (인라인 hex), `\d+px`가 컴포넌트에 직접 박힌 경우, `font-family:` 직접 선언.
발견 시 토큰/유틸리티로 교체하거나 해당 에이전트에 수정 요청한다.

## 핵심 원칙
- **토큰 우회 금지.** 모든 색·간격·폰트는 CSS 변수(또는 그것을 가리키는 Tailwind 유틸)를 거친다.
- **추정값은 추정으로 표기.** 검증 안 된 값에 거짓 확신을 부여하지 않는다.
- **단순함이 곧 진입장벽 낮춤.** 컴포넌트의 시각적 복잡도를 최소화한다.

상세 토큰 값·코드 스니펫·검증 체크리스트는 `references/wanted-tokens.md` 참조.
